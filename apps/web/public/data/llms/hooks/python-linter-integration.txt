## Hook Script
```bash
#!/bin/bash

# Read the tool input from stdin
INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.path // ""')

if [ -z "$FILE_PATH" ]; then
  exit 0
fi

# Check if this is a Python file
if [[ "$FILE_PATH" == *.py ]]; then
    echo "🔍 Python Linter Integration - Analyzing code quality..."
    echo "📄 File: $FILE_PATH"

    # Check if file exists
    if [ ! -f "$FILE_PATH" ]; then
        echo "⚠️ File not found: $FILE_PATH"
        exit 1
    fi

    # Check if pylint is available
    if command -v pylint >/dev/null 2>&1; then
        echo "🔍 Running pylint analysis..."

        # Run pylint with custom formatting
        PYLINT_OUTPUT=$(pylint "$FILE_PATH" --score=yes --reports=no --msg-template="{line:3d},{column:2d}: {category}: {msg} ({symbol})" 2>/dev/null)
        PYLINT_EXIT_CODE=$?

        if [ $PYLINT_EXIT_CODE -eq 0 ]; then
            echo "✅ Pylint analysis passed - No issues found"
        else
            echo "📋 Pylint Analysis Results:"
            echo "$PYLINT_OUTPUT" | head -20  # Limit output to first 20 lines

            # Show summary if there are many issues
            ISSUE_COUNT=$(echo "$PYLINT_OUTPUT" | wc -l)
            if [ "$ISSUE_COUNT" -gt 20 ]; then
                echo "... and $((ISSUE_COUNT - 20)) more issues"
            fi

            # Extract score if available
            SCORE=$(echo "$PYLINT_OUTPUT" | grep "Your code has been rated" | grep -o '[0-9]\+\.[0-9]\+' | head -1)
            if [ -n "$SCORE" ]; then
                echo "📊 Code Quality Score: $SCORE/10"
            fi
        fi

    elif command -v flake8 >/dev/null 2>&1; then
        echo "🔍 Running flake8 analysis (fallback)..."

        if flake8 "$FILE_PATH" --max-line-length=88; then
            echo "✅ Flake8 analysis passed - No style issues found"
        else
            echo "⚠️ Flake8 found style issues (non-blocking)"
        fi

    elif command -v pycodestyle >/dev/null 2>&1; then
        echo "🔍 Running pycodestyle analysis (fallback)..."

        if pycodestyle "$FILE_PATH" --max-line-length=88; then
            echo "✅ Pycodestyle analysis passed - No style issues found"
        else
            echo "⚠️ Pycodestyle found style issues (non-blocking)"
        fi

    else
        echo "⚠️ No Python linters found"
        echo "💡 Install options:"
        echo "  • pip install pylint (recommended)"
        echo "  • pip install flake8 (lightweight)"
        echo "  • pip install pycodestyle (basic)"
    fi

    # Additional code quality checks
    echo ""
    echo "📊 Quick Code Analysis:"

    # Count lines of code (excluding comments and empty lines)
    LOC=$(grep -v '^[[:space:]]*#' "$FILE_PATH" | grep -v '^[[:space:]]*$' | wc -l)
    echo "  • Lines of Code: $LOC"

    # Check for potential issues
    if grep -q "print(" "$FILE_PATH"; then
        echo "  • 💡 Consider using logging instead of print statements"
    fi

    if grep -q "TODO\|FIXME\|XXX" "$FILE_PATH"; then
        echo "  • 📝 TODOs/FIXMEs found - consider addressing them"
    fi

    # Check function complexity (rough estimate)
    FUNCTION_COUNT=$(grep -c "^def " "$FILE_PATH")
    if [ "$FUNCTION_COUNT" -gt 0 ]; then
        AVG_LINES_PER_FUNC=$((LOC / FUNCTION_COUNT))
        echo "  • Functions: $FUNCTION_COUNT (avg ~$AVG_LINES_PER_FUNC lines each)"

        if [ "$AVG_LINES_PER_FUNC" -gt 20 ]; then
            echo "  • 💡 Consider breaking down large functions"
        fi
    fi

    echo ""
    echo "💡 Python Code Quality Tips:"
    echo "  • Follow PEP 8 style guidelines"
    echo "  • Use meaningful variable and function names"
    echo "  • Add docstrings to functions and classes"
    echo "  • Keep functions small and focused"
    echo "  • Use type hints for better code clarity"

    echo ""
    echo "🎯 Code quality analysis complete!"

else
    echo "ℹ️ File is not a Python file: $FILE_PATH"
fi

exit 0
```

#!/bin/bash

      fi
    fi
  fi
fi

### Multi-Linter Support with Fallback

Enhanced hook script with multi-linter support and automatic fallback

```bash
#!/bin/bash
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.path // ""')
if [[ "$FILE_PATH" == *.py ]]; then
  if command -v pylint >/dev/null 2>&1; then
    echo "🔍 Running pylint analysis..."
    pylint "$FILE_PATH" --score=yes --reports=no
  elif command -v flake8 >/dev/null 2>&1; then
    echo "🔍 Running flake8 analysis (fallback)..."
    flake8 "$FILE_PATH" --max-line-length=88
  elif command -v ruff >/dev/null 2>&1; then
    echo "🔍 Running ruff analysis (fallback)..."
    ruff check "$FILE_PATH"
  elif command -v pycodestyle >/dev/null 2>&1; then
    echo "🔍 Running pycodestyle analysis (fallback)..."
    pycodestyle "$FILE_PATH" --max-line-length=88
  else
    echo "⚠️ No Python linters found"
  fi
fi
exit 0
```

## Troubleshooting
### Hook reports no linters found despite installation

Ensure linters are in your PATH. Use 'which pylint' or 'which flake8' to verify. Install globally with 'pip install pylint' or activate your virtual environment before running hooks. Verify Python environment and PATH. Test with various Python environments.

### Pylint output is truncated showing only first 20 lines

This is intentional to prevent overwhelming output. For full analysis, run 'pylint <file>' directly. The hook shows summary statistics including total issue count and quality score. Adjust head -20 limit if needed. Test with various file sizes.

### Hook shows warnings but doesn't block file operations

This is by design - linting runs post-modification as feedback. To block writes, move the hook to preToolUse and add 'exit 1' for failing scores. Consider your workflow before enforcing strict blocks. Verify hook configuration. Test with various linting scenarios.

### False positives or style conflicts with project standards

Create a .pylintrc or .flake8 config file in your project root to customize rules. Use '--disable=<rule>' for specific suppressions. The hook respects project-level configuration files. Verify configuration files. Test with various rule configurations.

### Virtual environment packages cause import errors in pylint

Ensure the hook runs in the same environment where Python packages are installed. Set 'export VIRTUAL_ENV=/path/to/venv' or activate venv before Claude Code session starts. Verify virtual environment activation. Test with various virtual environment configurations.

### Pylint is too slow on large Python files

Consider using ruff as a faster alternative (ruff check). Configure pylint with --jobs=0 for parallel processing. Exclude large files or use incremental linting. Verify file size limits. Test with various file sizes.

### Multiple linters show conflicting results

Different linters have different rule sets and philosophies. Choose one primary linter (pylint recommended) and use others as fallbacks. Configure linters consistently with project standards. Verify linter configurations. Test with various linter combinations.

### Code quality score is low but code works correctly

Pylint scoring considers maintainability, not just correctness. Review specific issues and address high-priority ones. Use --disable flags for false positives. Focus on improving maintainability over time. Verify code quality metrics. Test with various code quality scenarios.
